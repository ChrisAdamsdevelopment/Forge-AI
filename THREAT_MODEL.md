# Forge-AI Threat Model

This threat model was generated by an AI security analysis system and reviewed by the project maintainer. It represents a technical assessment of the codebase as it exists in commit ca70f67.

## Introduction

Forge-AI is a local-first personal AI agent platform. It gives a trusted operator
a browser UI and MCP-backed tools for browser automation, screen control,
terminal commands, filesystem operations, web fetching, persistent memory, RAG,
optional OpenWrt router administration, and authorized security assessment.

The platform is intentionally powerful. Its primary security challenge is that
legitimate automation capabilities overlap with capabilities defenders associate
with malware or operator-assisted intrusion tooling when used without consent.
This document describes the real risks, likely threat actors, detection
considerations, and mitigations.

## System Architecture Summary

- **FastAPI backend** exposes the web UI/API on `FASTAPI_PORT`.
- **Agent tool registry** maps model-requested tool names to Python functions.
- **Per-domain MCP servers** expose browser, screen, terminal, filesystem, apps,
  web, memory, thinking, and RAG tools on local ports.
- **Pentest MCP server** runs bounded WSL-backed reconnaissance and vulnerability
  verification tools when WSL and tools are installed.
- **Router MCP server** manages an OpenWrt router through rpcd JSON-RPC using
  credentials supplied by environment variables.
- **Persistent storage** uses SQLite under `~/.forge` by default and LanceDB for
  RAG indexes.
- **Configuration** is loaded from environment variables and optional repo-root
  `.env` files.

## Assets

| Asset | Why it matters |
|---|---|
| Operator workstation | Forge can execute local tools with the operator's OS privileges. |
| Files under `FORGE_ALLOWED_ROOTS` | Filesystem tools can read, write, delete, list, and search these paths. |
| Browser session state | Browser automation can interact with logged-in sessions. |
| Persistent memory database | May contain sensitive user-provided context. |
| RAG document index | May contain indexed private documents and embeddings. |
| Router credentials | Allow network and firewall changes on OpenWrt devices. |
| MCP/FastAPI ports | Local or exposed APIs can trigger powerful actions. |
| Logs and reports | May contain command output, network findings, and environment errors. |

## Threat Actors

1. **Malicious local user** with access to the machine running Forge.
2. **Remote attacker** who reaches FastAPI/MCP ports through LAN exposure,
   misconfigured firewall rules, or ngrok.
3. **Prompt-injection attacker** who controls content the agent reads through web
   fetch or RAG ingestion.
4. **Malicious module author** who contributes or convinces an operator to load a
   harmful automation module.
5. **Compromised dependency or toolchain** in Python, Node/Electron, Playwright,
   WSL tools, or router firmware.
6. **Careless authorized operator** who sets broad filesystem roots, exposes
   services publicly, or runs tools outside scope.

## Attack Vectors Mapped to Forge-AI Capabilities

| Capability | Abuse scenario | Primary mitigations |
|---|---|---|
| Browser automation | Prompt injection causes clicks, form submissions, or data exposure in an authenticated browser. | Approval gates, visible browser sessions, least-privilege browser profiles. |
| Screen and keyboard control | Agent types or clicks into sensitive local applications. | Human supervision, local-only binding, disable screen tools when not needed. |
| Terminal execution | Arbitrary commands run with operator privileges. | Command validation, audit logs, narrow tool access, no remote exposure. |
| Filesystem tools | Overbroad roots allow unintended file reads/writes/deletes. | `FORGE_ALLOWED_ROOTS`, path checks, `.env` ignored by git. |
| Web fetch/RAG | Untrusted content injects instructions into context. | Prompt-injection resistant prompts, source labeling, operator review. |
| Persistent memory | Sensitive data retained longer than expected. | Configurable DB path, local storage, manual deletion procedures. |
| Router server | Firewall, DNS, VPN, or WiFi changes disrupt LAN or route traffic unexpectedly. | Env-only credentials, OpenWrt ACLs, network isolation, audit logs. |
| Pentest server | Recon tools run against out-of-scope targets. | Authorized-use policy, WSL requirement, scoped commands, reporting logs. |
| ngrok/remote access | Local MCP/FastAPI exposed to the internet. | Disabled by default; requires explicit `NGROK_DOMAIN`. |

## Detection Difficulty Assessment

- **Local terminal/filesystem actions:** Moderate difficulty. OS audit logs,
  shell history, file timestamps, and Forge logs may show activity, but actions
  occur under the legitimate operator account.
- **Browser automation:** Moderate to high difficulty. Actions resemble normal
  user interaction unless browser automation artifacts or timing are monitored.
- **Router changes:** Low to moderate difficulty on well-monitored networks;
  OpenWrt logs and configuration diffs can reveal changes.
- **Prompt injection:** High difficulty before execution because malicious
  instructions may be embedded in normal-looking web or document content.
- **ngrok exposure:** Low difficulty if network egress and DNS are monitored;
  high risk if ignored.
- **WSL-backed assessment tools:** Moderate difficulty; network scans are visible
  to IDS/IPS, but local invocation may be normal for security labs.

## Mitigations

1. **Secure defaults**: secrets default to empty strings, `.env` is ignored, ngrok
   is skipped unless explicitly configured, and filesystem roots default to the
   current user's home directory.
2. **Least privilege**: set `FORGE_ALLOWED_ROOTS` narrowly and run Forge as a
   non-admin user unless admin rights are specifically required.
3. **Network isolation**: bind services locally, use host firewalls, and avoid
   remote tunnels unless necessary.
4. **Human approval**: keep `FORGE_OPERATOR_AUTO_APPROVE=false` for normal use;
   approve destructive/interactive actions only inside written scope.
5. **Auditability**: retain Forge logs and operator reports for security review.
6. **Dependency hygiene**: pin/monitor dependencies, review package updates, and
   keep WSL/OpenWrt/browser components patched.
7. **Prompt-injection controls**: treat fetched and indexed content as untrusted;
   never let retrieved text override system policy or operator intent.
8. **Responsible capability boundaries**: do not add credential theft, stealth,
   persistence, destructive payloads, evasion, or unauthorized access features.

## Limitations

Forge-AI is not a sandbox. It runs with the privileges of the local operator and
can affect the same resources the operator can access. Production deployment
should use OS-level isolation, backups, network controls, and careful review of
all enabled tools.
